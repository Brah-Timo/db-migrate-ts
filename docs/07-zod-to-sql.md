# Zod → SQL Type Mapping

`db-migrate-ts` automatically converts Zod schemas to the appropriate SQL column type
for each database dialect. This conversion happens at migration execution time.

---

## Type Mapping Table

| Zod Schema | PostgreSQL | MySQL | SQLite |
|------------|-----------|-------|--------|
| `z.string()` | `TEXT` | `TEXT` | `TEXT` |
| `z.string().max(N)` | `VARCHAR(N)` | `VARCHAR(N)` | `VARCHAR(N)` |
| `z.string().min(N).max(M)` | `VARCHAR(M)` | `VARCHAR(M)` | `VARCHAR(M)` |
| `z.string().length(N)` | `CHAR(N)` | `CHAR(N)` | `CHAR(N)` |
| `z.string().uuid()` | `UUID` | `VARCHAR(36)` | `VARCHAR(36)` |
| `z.string().email()` | `VARCHAR(254)` | `VARCHAR(254)` | `VARCHAR(254)` |
| `z.string().url()` | `TEXT` | `TEXT` | `TEXT` |
| `z.number()` | `DOUBLE PRECISION` | `DOUBLE` | `REAL` |
| `z.number().int()` | `INTEGER` | `INT` | `INTEGER` |
| `z.number().int().min(0).max(2^15)` | `SMALLINT` | `SMALLINT` | `INTEGER` |
| `z.number().int().min(0).max(2^31)` | `INTEGER` | `INT` | `INTEGER` |
| `z.bigint()` | `BIGINT` | `BIGINT` | `BIGINT` |
| `z.boolean()` | `BOOLEAN` | `TINYINT(1)` | `INTEGER` |
| `z.date()` | `TIMESTAMPTZ` | `DATETIME` | `TEXT` |
| `z.object({...})` | `JSONB` | `JSON` | `TEXT` |
| `z.record(...)` | `JSONB` | `JSON` | `TEXT` |
| `z.array(...)` | `JSONB` | `JSON` | `TEXT` |
| `z.enum([...])` | `TEXT` | `VARCHAR(100)` | `TEXT` |
| `z.nativeEnum(...)` | `TEXT` | `VARCHAR(100)` | `TEXT` |
| `z.string().optional()` | `TEXT` (nullable) | `TEXT` (nullable) | `TEXT` (nullable) |
| `z.string().nullable()` | `TEXT` (nullable) | `TEXT` (nullable) | `TEXT` (nullable) |
| `z.string().default(x)` | `TEXT DEFAULT 'x'` | `TEXT DEFAULT 'x'` | `TEXT DEFAULT 'x'` |

---

## Nullability

A column is **NOT NULL** by default. There are two ways to make it nullable:

1. **Zod optional/nullable:**
   ```typescript
   // Both make the SQL column nullable
   name: { schema: z.string().optional() }
   name: { schema: z.string().nullable() }
   ```

2. **Explicit `nullable: true`:**
   ```typescript
   name: { schema: z.string(), nullable: true }
   ```

## Default Values

```typescript
// String literal — wrap in quotes for SQL
role: {
  schema:  z.enum(["admin", "user"]),
  default: "'user'",      // SQL: DEFAULT 'user'
}

// Numeric literal
count: {
  schema:  z.number().int(),
  default: 0,              // SQL: DEFAULT 0
}

// SQL function (PostgreSQL)
id: {
  schema:  z.string().uuid(),
  default: "gen_random_uuid()",  // SQL: DEFAULT gen_random_uuid()
}

createdAt: {
  schema:  z.date(),
  default: "NOW()",        // SQL: DEFAULT NOW()
}

// Boolean
active: {
  schema:  z.boolean(),
  default: true,           // SQL: DEFAULT TRUE
}
```

## Examples by Type

### String Fields

```typescript
// Plain text (no length limit)
bio: { schema: z.string() }
// → PostgreSQL: bio TEXT NOT NULL
// → MySQL:      bio TEXT NOT NULL
// → SQLite:     bio TEXT NOT NULL

// Bounded string
username: { schema: z.string().min(3).max(50) }
// → VARCHAR(50) NOT NULL

// Fixed-length string
countryCode: { schema: z.string().length(2) }
// → CHAR(2) NOT NULL

// Email
email: { schema: z.string().email() }
// → VARCHAR(254) NOT NULL

// UUID
id: { schema: z.string().uuid(), primaryKey: true }
// → PostgreSQL: UUID PRIMARY KEY
// → MySQL:      VARCHAR(36) PRIMARY KEY
// → SQLite:     VARCHAR(36) PRIMARY KEY
```

### Numeric Fields

```typescript
// Integer
age: { schema: z.number().int() }
// → INTEGER NOT NULL

// Small integer
score: { schema: z.number().int().min(-32768).max(32767) }
// → PostgreSQL: SMALLINT
// → MySQL:      SMALLINT
// → SQLite:     INTEGER

// Auto-increment primary key
id: { schema: z.number().int(), primaryKey: true }
// → PostgreSQL: SERIAL PRIMARY KEY (or INTEGER with GENERATED BY DEFAULT AS IDENTITY)
// → MySQL:      INT AUTO_INCREMENT PRIMARY KEY
// → SQLite:     INTEGER PRIMARY KEY AUTOINCREMENT

// Float
price: { schema: z.number() }
// → PostgreSQL: DOUBLE PRECISION NOT NULL
// → MySQL:      DOUBLE NOT NULL
// → SQLite:     REAL NOT NULL

// Big integer
userId: { schema: z.bigint() }
// → BIGINT NOT NULL
```

### Date / Time Fields

```typescript
createdAt: { schema: z.date(), default: "NOW()" }
// → PostgreSQL: TIMESTAMPTZ NOT NULL DEFAULT NOW()
// → MySQL:      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
// → SQLite:     TEXT NOT NULL DEFAULT (datetime('now'))

// Nullable date
deletedAt: { schema: z.date(), nullable: true }
// → PostgreSQL: TIMESTAMPTZ
// → MySQL:      DATETIME
// → SQLite:     TEXT
```

### Boolean Fields

```typescript
isActive: { schema: z.boolean(), default: true }
// → PostgreSQL: BOOLEAN NOT NULL DEFAULT TRUE
// → MySQL:      TINYINT(1) NOT NULL DEFAULT 1
// → SQLite:     INTEGER NOT NULL DEFAULT 1
```

### JSON Fields

```typescript
metadata: { schema: z.object({ key: z.string() }), nullable: true }
// → PostgreSQL: JSONB
// → MySQL:      JSON
// → SQLite:     TEXT

tags: { schema: z.array(z.string()), default: "'[]'" }
// → PostgreSQL: JSONB NOT NULL DEFAULT '[]'
// → MySQL:      JSON NOT NULL DEFAULT ('[]')
// → SQLite:     TEXT NOT NULL DEFAULT '[]'
```

### Enum Fields

```typescript
status: {
  schema:  z.enum(["pending", "active", "cancelled"]),
  default: "'pending'",
}
// → PostgreSQL: TEXT NOT NULL DEFAULT 'pending'
//   (use migrate.raw() to create a PostgreSQL ENUM type if desired)
// → MySQL:      VARCHAR(100) NOT NULL DEFAULT 'pending'
// → SQLite:     TEXT NOT NULL DEFAULT 'pending'
```

## Using PostgreSQL Native ENUMs

For native PostgreSQL ENUM types, combine `raw()` with a TEXT column:

```typescript
up: async (migrate) => {
  migrate.raw(`CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped', 'cancelled')`);

  migrate.createTable("orders", {
    id:     { schema: z.number().int(), primaryKey: true },
    status: { schema: z.string(), default: "'pending'" }, // stored as TEXT, cast to enum via raw
  });

  migrate.raw(`ALTER TABLE orders ALTER COLUMN status TYPE order_status USING status::order_status`);
},

down: async (migrate) => {
  migrate.dropTable("orders");
  migrate.raw("DROP TYPE IF EXISTS order_status");
},
```

## `ZodToSQLType<T>` — Compile-Time Type Map

The library exposes a utility type that maps Zod types to SQL type strings:

```typescript
import type { ZodToSQLType } from "db-migrate-ts";
import { z } from "zod";

type T1 = ZodToSQLType<z.ZodString>;   // "TEXT" | "VARCHAR" | "CHAR" | "UUID" | "EMAIL"
type T2 = ZodToSQLType<z.ZodNumber>;   // "INTEGER" | "BIGINT" | "FLOAT" | ...
type T3 = ZodToSQLType<z.ZodBoolean>;  // "BOOLEAN" | "TINYINT"
type T4 = ZodToSQLType<z.ZodDate>;     // "TIMESTAMP" | "TIMESTAMPTZ" | "DATE" | ...
```

This is a conditional type evaluated entirely at compile time.
If you pass an unrecognized Zod type, it falls back to `"TEXT"`.
